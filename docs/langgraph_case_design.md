# LangGraph Agent Case Design

## Design Goal

The benchmark represents the Agent workload Agentrix is designed for: a long
shared evidence/history root produces many concurrent, role-specific reasoning
branches, each can call a real RAG tool, and a reducer consumes their outputs.
It is intentionally different from batching unrelated chat prompts or copying
one synthetic token string into many requests.

The case set must exercise three reuse relations:

- within one case, sibling branches inherit an exact long parent prefix;
- a branch retrieval may return evidence already visible in that parent;
- across cases, RAG documents can recur in a different order.

These relations target ForkAttention, application compaction, and CacheBlend
respectively without requiring one mechanism to impersonate another.

## Live Graph

Each case executes the following StateGraph:

```text
START
  -> retrieve_context (deterministic local BM25, 24 chunks)
  -> planner (live vLLM request)
  -> fanout 16 branch_agent nodes
       -> tool selection (live vLLM function call)
       -> rag_search (real local tool execution)
       -> optional exact duplicate-chunk compaction
       -> branch reflection (live vLLM request)
  -> reducer (live vLLM request over all branch answers)
  -> END
```

Planner output, generated tool query, tool result, branch answer, and reducer
input propagate through the graph. There are no mocked LLM responses, fixed
sleep-based inference times, web-search calls, or hidden precomputed answers.

## Exact Shared-Root Construction

The planner receives:

1. a stable system instruction;
2. the bootstrap RAG evidence;
3. the task;
4. the common-analysis request.

Every branch tool-selection request begins with those exact messages followed
by the planner's assistant response. Only then is its private role instruction
added. The normal planner call therefore warms the same prefix inherited by
all branches; there is no extra unmeasured copy of the final branch prompt.

Each case has 16 branches. Eight primary roles cover independent technical or
operational perspectives; the other eight are explicit independent
verification copies of those roles. This is realistic for multi-agent review
and maps to the maximum 16-query shared cohort of the evaluated Qwen3 GQA
shape. The role suffix prevents branches from being identical requests.

At most one independent case is active, while its 16 branches are concurrent.
This case-major policy models one active Agent workflow fanning out internally
and keeps its siblings resident together. The earlier mixed-queue experiment
is retained as a scheduling negative control: many overlapping unrelated cases
can eliminate ForkAttention benefit even when every prompt is long.

## RAG Corpus and Retrieval

`LocalRAG` indexes real repository documentation rather than generated filler.
It reads Markdown, Python, or text files listed in
`benchmark/configs/langgraph_rag_corpus_files.txt`, splits them into
1,800-character source chunks, and returns up to 1,200 characters per result.

Every chunk identity includes:

- the relative source path;
- its character offset;
- a 12-hex SHA-256 content digest.

The manifest fixes membership to 11 source files. The formal experiment used
101 chunks with corpus version `6f2b83c2680700dd`; after adding the documented
results to included sources, the current version is `b01c0c10bb027921`.
The runner checks this content-addressed version and fails closed on drift.
Result and overview documents are excluded, preventing self-reference and
benchmark leakage.

The bootstrap query is a case-authored topic query used to establish broad
shared evidence. Each branch's focused query is generated by Qwen through the
`rag_search` tool schema, with `top_k` constrained to 1-5. The tool returns
source paths and original chunk text. Scores are not inserted into stable RAG
segments because scores vary with the query and would defeat exact segment
identity.

## Case Shape

| Property | Value | Reason |
|---|---:|---|
| Distinct cases | 20 | Covers five Agent system domains without repeating tasks |
| Cases per domain | 4 | Balanced topic coverage |
| Bootstrap chunks | 24 | Produces an 8K-or-longer shared parent naturally |
| Bootstrap cap | 32,000 characters | Bounds context while retaining long evidence |
| Branches per case | 16 | Strong ForkAttention cohort and broad review |
| Active independent cases | 1 | Case-major production fanout positive control |
| Branch tool | Local `rag_search` | Real deterministic tool, no network variance |
| Branch output budget | 256 tokens | Keeps branches alive long enough to share decode |
| Reducer budget | 192 tokens | Exercises downstream aggregation |

Measured tool-selection prompts are approximately 8.8K-10.2K tokens, with a
median near 9.36K. The shared parent dominates each short private role suffix,
placing the workload inside the validated ForkAttention target region without
padding a fixed token count.

## The Twenty Tasks

| Domain | Case IDs | Agent purpose |
|---|---|---|
| Fork performance | `fork-graph-regression`, `fork-tail-latency`, `fork-flash-crossover`, `fork-kernel-audit` | Diagnose graphs, CTA waste, fallback, crossover, and kernel evidence |
| RAG and cache | `rag-cacheblend-design`, `rag-tiered-offload`, `rag-quality-risk`, `rag-cache-capacity` | Design reordered-chunk reuse, tiers, quality gates, and capacity |
| DP scheduling | `dp-prefix-routing`, `dp-graph-capacity`, `dp-kv-residency`, `dp-arrival-waves` | Balance load, affinity, graph capacity, residency, and arrivals |
| Deployment | `deploy-cuda-host`, `deploy-memory-failure`, `deploy-observability`, `deploy-release-gate` | Review environment, memory failures, telemetry, and release risk |
| Portability | `port-metal-support`, `port-musa-build`, `port-centos-hce`, `port-tp-correctness` | Assess Metal, MUSA, Linux toolchains, TP support, and correctness |

The tasks ask for diagnosis, design, or review grounded in the local corpus.
They are distinct but deliberately share system documents. This mirrors a
team of Agents working on related parts of one codebase: repeated policy and
architecture evidence is natural, while the requested conclusions and branch
roles remain different.

## How Each Optimization Gets a Legitimate Opportunity

### ForkAttention

All branch requests inherit the exact bootstrap, task, and planner response.
Sixteen siblings are admitted together, APC keeps physical prefix blocks
resident, and branch generation is long enough for synchronized decode. The
benchmark records physical ForkAttention active-step and CTA counters, so a
logical tree alone cannot be mistaken for kernel execution.

### Application Compaction

A focused branch search often returns one of the 24 bootstrap chunks. When
compaction is enabled, only the repeated copy in the later tool message is
removed. The shared root, new tool chunks, tool-call protocol, and original
text remain. Stable IDs with different bytes raise an error instead of being
merged.

### CacheBlend

CacheBlend-formatted evidence places stable separators around documents. The
20 bootstraps contain 480 chunk occurrences but 83 unique chunks; 82.81% of
their characters are reusable, and 184 case pairs reorder shared evidence.
This is the intended reordered-RAG pattern. A high lookup hit rate is verified
from LMCache logs rather than assumed from overlap.

## Realism, Fairness, and Limits

The benchmark is realistic in control flow and serving behavior, but it is not
a claim to reproduce every production queue:

- the knowledge base is local Agentrix documentation, not a production vector
  database;
- BM25 retrieval is deterministic and tool latency is near zero;
- web search was deliberately removed to avoid network and freshness noise;
- one active case is a ForkAttention positive-control admission policy;
- Qwen3-0.6B is useful for system measurement but weak for semantic scoring;
- tasks have no executable ground-truth outcome or task-specific grader.

Fair comparisons keep the task file, manifest, model, token budgets, case
order, and warm-up fixed and restart the server per variant. Baseline and Fork
match APC/graph settings. CacheBlend is reported as a separate deployment path
because it requires eager execution and disabled APC. It is disabled by
default and requires `ENABLE_CACHEBLEND=1` in the benchmark launchers.

Live execution is necessary to show Agent relevance, but it permits output
drift to alter later requests. A complete evaluation therefore uses two
layers:

1. live runs for workflow completion, semantic evaluation, and operational
   realism;
2. raw/compacted fixed-trace replay for strict latency and memory attribution.

Valid tool-call rate and non-empty reducer rate are execution gates. Lexical F1
is a drift diagnostic. Neither replaces a domain rubric, citation check, or
task-success evaluator.
